# Design Decisions

Decisions made and rationale. Last updated: 2026-02-22.

---

## 1. Canonical packaging: Clawhub-installable Expert Plugins

**Decision:** Expert Plugins should eventually be installable via Clawhub as a first-class
distribution unit. If the registry supports a "Roles" type separately from "Skills",
use that. Otherwise, Skills is the vehicle.

**Rationale:** Expert Plugins are shareable, reusable domain bundles. The same distribution
mechanism that makes Skills composable should apply to Roles.

---

## 2. Main session injection: honest context-append

**Decision:** Accept the honest limitation — role injection in the main session is
context-append only. There is no true unload. Document this clearly; do not pretend
otherwise.

**Rationale:** Pretending to unload while leaving the context in the session is
misleading and causes unpredictable behaviour. Honest design is better than a
convincing illusion.

---

## 3. Unload semantics: "start a new thread"

**Decision:** "Unload role" means "start a new thread / session." This is the honest
definition. Not a pretend-unload.

**Rationale:** A new thread is a genuinely clean context. It's the only mechanism that
actually removes the injected role from the active context.

---

## 4. Practitioner naming convention

**Decision:** Expert Plugins are named after the human practitioner of the same role.
"Mandarin Teacher", "Health Coach", "Research Analyst", "Executive Assistant" — not
"Mandarin", "Health", "Research", "Executive".

**Rationale:** Practitioner names are *role titles*, not tool names or feature flags.
"Health Coach" sounds like a person you'd hire. "Health" sounds like a toggle.
The former sets the right mental model — you're loading a collaborator with domain
expertise. The naming convention reinforces this.

---

## 5. Expert Plugin file structure

**Decision:** Each Expert Plugin contains three files/directories:

```
~/clawd/experts/[practitioner-name]/
├── EXPERT.md              ← domain expertise + personalised context + SOPs
├── skills/          ← how this practitioner stays current
└── resources/
    └── INDEX.md         ← curated reference library with usage guidance
```

**EXPERT.md** — unchanged from original design. Domain knowledge, SOPs, skill pointers,
personalised context, data sources, constraints.

**skills/** — new. Journals, blogs, newsletters, podcasts, Twitter accounts,
subreddits, YouTube channels, tools to watch. How a practitioner in this domain stays
current. Feeds automated digests and active research tasks.

**resources/** — new. A bookshelf. Reference materials the practitioner would have:
e-books, papers, guides, quick-reference docs. `INDEX.md` tells the agent what's in
the library and when to reach for each item. Files are loaded lazily — on role load,
read INDEX.md; pull individual resources when the task calls for it.

---

## 6. Role creation: role-creator skill

**Decision:** Building new Expert Plugins is itself a formalised workflow — the
`role-creator` skill. It lives in `~/clawd/skills/role-creator/` (a Skill, not a
Expert Plugin — it's a factory tool, not a practitioner).

**Rationale:** The role-creator is a workflow tool. A hammer isn't a craftsman.

**Process:** 4 phases — Domain Research (autonomous), Intake (interactive, ~10 min),
Generation (autonomous), Calibration (interactive, ~10 min).

See: `~/clawd/skills/role-creator/SKILL.md`

---

## 7. Skills vs Expert Plugins distinction

**Decision:** Keep Skills and Expert Plugins as separate concepts with separate directories:
- `~/clawd/skills/` — tools and capabilities (perplexity, gog, spotify-full, role-creator)
- `~/clawd/experts/` — practitioner bundles (health-coach, mandarin-teacher, research-analyst)

**Rationale:** Skills are tools you use. Expert Plugins are expertise you embody. The
metaphor and the mechanics are different.

---

## 8. Nomenclature: "Onboarding" not "Intake"

**Decision:** The personalisation process is called "Onboarding." The file is
`ONBOARDING.md` (not `INTAKE.md`).

**Rationale:** "Intake" carries medical connotations. "Onboarding" is domain-neutral,
widely understood, and fits the practitioner metaphor — you onboard into a new role.

---

## 9. Published vs local split (USER.md pattern)

**Decision:** Adopt the `.local` naming convention from software development.

**Published (ships with the Expert Plugin, shareable on Clawhub):**
- `EXPERT.md` — universal domain expertise + generic SOPs
- `ONBOARDING.md` — questions + instructions for generating personalisation
- `skills/` — ongoing sources
- `resources/INDEX.md` — reference library index

**Local only (gitignored, never shared, private to the user):**
- `USER.md` — personalised context, generated by onboarding
- `resources/*.pdf` etc. — user-uploaded materials

**Reference:** Anthropic's Legal plugin (github.com/anthropics/knowledge-work-plugins)
uses the same pattern: generic plugin + `legal.local.md` for org-specific positions.

**Behaviour:**
- On role load: read EXPERT.md + USER.md (if exists)
- If USER.md absent: run ONBOARDING.md flow → generate USER.md
- USER.md is human-readable and directly editable — not agent-only
- Re-onboard: `mv ONBOARDING.complete ONBOARDING.md` and re-run
